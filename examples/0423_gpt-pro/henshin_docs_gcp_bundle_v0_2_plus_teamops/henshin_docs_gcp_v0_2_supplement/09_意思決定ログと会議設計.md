# 09_意思決定ログと会議設計

Version: 0.2.1  
用途: 強いエンジニアに引っ張られすぎず、素人PMが置いていかれない会議体を作る

---

## 1. 必要な会議は4種類だけ
### M1. Product Decision
レイヤー:
- L0 ロア / 体験
- L1 一部のデータ契約

決めること:
- 何を作るか
- 何を作らないか
- 受け入れ基準

参加者:
- PM
- Lead Engineer
- 必要なら XR / Web 代表1名ずつ

頻度:
- 週1

### M2. Architecture / Data Contract
レイヤー:
- L1 / L2

決めること:
- SuitManifest
- PartCatalog
- Firestore / SQL split
- API責務

参加者:
- Lead Engineer
- Backend
- XR
- PM（決定確認役）

頻度:
- 週1〜2

### M3. Delivery / Implementation Sync
レイヤー:
- L3

決めること:
- 今週のタスク
- ブロッカー
- 実装順

参加者:
- 実装担当
- PMはオプション

頻度:
- 週2

### M4. AI Output Review
レイヤー:
- 差分確認

決めること:
- AI提案の採用/保留/却下
- 正本に反映するか

参加者:
- AI担当
- 関係Lead
- PM

頻度:
- 必要時

---

## 4. 会議で絶対に記録すべき項目
- 決定
- 保留
- 誰が決めるか
- 何を見ればその話が分かるか
- 次回までの宿題

---

## 5. PM保護ルール
- PMが理解できないまま「それでいきましょう」は禁止
- 理解できないときは「3行で言い換えて」を要求してよい
- 図が必要なときは、図を宿題にして会議を閉じてよい
- その場で全部理解しようとしない。用語は glossary に落とす

---

## 6. エンジニア保護ルール
- 仕様未確定の実装見積もりは “仮” と明記
- PMのロア話が広がったら、M1へ戻す
- UIモックの話で Data Contract を変えない
- 「あとで直せる」ではなく「どこが壊れるか」で説明する

---

## 7. 今のフェーズでの会議のおすすめ順序
1. Product Decision: Replay と Trial の境界確認

---

## 8. 一文でいうと
**会議は“情報共有の場”ではなく、“何を決め、何を保留にするかを記録する場”にする。**
