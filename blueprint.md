1.1.1 コンセプト更新：令和ギャバンは「エモルギー」で蒸着する

ここで言うエモルギーは、感情そのものをそのまま正義や資格にする話ではない。  

**感情 = 原料、誓い = 安定化条件、蒸着 = 制度化** と分けて扱う。

- エモルギー：感情から生じる未安定の蒸着原料
- エモルギア：計測・封止され、再利用可能になったエモルギー単位
- 誓い：その原料を暴走させず、正規スーツとして成立させる規範
- スーツ：感情をそのまま着るのではなく、感情を制度に通して法定状態へ変えた結果

この分離をやると、現行ロアの「誓いは行為規範」を壊さずに、令和らしい“感情駆動の熱”を入れられる。

1.1.2 生成方針：全部生成ではなく「固定骨格 + 可変意匠」

全部を生成にすると、毎回かっこよくてもスーツとしての同一世界観が崩れる。  

勝ち筋は次のハイブリッド：

- 固定するもの：感情色相マップ、モジュール構成、シルエット系統、蒸着ルール、封印ルール
- 半固定にするもの：感情ベクトル→スーツ要素への変換表
- 生成に任せるもの：青写真、エンブレム、表面柄、イラスト、蒸着エフェクトの差分

つまり、**「骨格はルールベース、見た目の熱量は生成」**でいく。

3.3 EmolgiaSeed（感情→スーツ変換の中間データ）
目的：入力文や選択色を、そのまま画像生成へ投げず、まず“安定した設計入力”へ圧縮する。

仮の6軸：

- 高揚：黄-橙 / 放射 / 上向き拡張 / 火花型蒸着
- 闘志：赤 / 直線 / 前方突出 / スパーク集中型蒸着
- 哀傷：青-藍 / 垂直落下 / 細長い後方残光 / ミスト収束型蒸着
- 緊張：紫-黒 / 細線密集 / 隙間のある多層 / ノイズ抑制型蒸着
- 守護：緑-青緑 / 面で包む / 胸と肩を厚く / シールド展開型蒸着
- 受容：桃-紅 / 曲線 / 外縁保護 / 花弁拡散型蒸着

重要：`SuitSpec` にいきなり生の感情文を入れない。

4.1 Track A：Suit Forge（SIM & 生成）仕様
A-1. 鎧立て（アーマースタンド）＝最初の快感ポイント
入力：感情テキスト or キーワード + 誓い（内部用） + 色 + 形状（スライダーでもOK） + 任意のラフ絵

MVPでは次の3段構えが安定：

- 主感情を1つ選ぶ
- 副感情を1つ選ぶ
- 補助として自由文を添える

これで「1:1対応の気持ち良さ」と「崩れにくさ」の両方を取りやすい。

出力：

SuitSpec.json

Gemini 3系の画像生成は「画像生成・編集」を扱い、会話的な編集では thoughtSignature が重要と説明されている。 (Google AI for Developers) → 3面図の整合性を上げるために「最初の1枚→編集で側面/背面」など、反復編集を前提にしたパイプラインを検討する価値がある（最初から完璧な3面図を一発で狙わない）。
A-1.1 感情色相マップ（最初に固定するルール）

最初に「感情名 ↔ 色相帯 ↔ 線の性格 ↔ 表面モチーフ」の対応表を作る。  

今回は試しに 6軸で固定する。  

- 高揚：黄-橙系、放射線、発光面多め、頭部クラウン上振れ
- 闘志：赤系、直線・尖り、前方へ張り出す、肩と前腕が大型化
- 哀傷：青-藍系、落ちる線、長い残光、脚部が細長く伸びる
- 緊張：紫-黒系、細線密集、隙間の多い装甲、面数が増えて密度が上がる
- 守護：緑-青緑系、面で包む、厚めの胸郭、胸と肩のシェルが拡張
- 受容：桃-紅系、曲線、包み込む外縁、腰回りと背面の保護輪郭が広がる

この6軸でまず運用し、曖昧さが強ければ 7-8軸化する。

A-1.1.1 蒸着エフェクトは「色相帯で半固定」にする

蒸着エフェクトまで毎回完全生成にすると、変身の記号性が壊れやすい。  

なので、蒸着は **感情軸ごとの固定プロファイル** を持ち、個別差分は強度と制御性で微調整する。

例：

- 高揚：明滅間隔が短い、火花が上方向へ抜ける
- 闘志：収束が速い、中心から外へスパークが走る
- 哀傷：粒子が一度散ってから静かに定着する
- 緊張：細い線分が多数走り、最終定着が遅い
- 守護：外周から内側へシールド状に閉じる
- 受容：胸部中心からやわらかく広がって包み込む

ここは「演出の個性」ではなく「変身の文法」として固定してよい。

A-1.2 1:1対応の変身ロジック

1. 入力文を感情ベクトルへ落とす  
2. 感情ベクトルを EmolgiaSeed へ圧縮する  
3. EmolgiaSeed から `palette / motif / silhouette_family / silhouette_params / deposition_profile / emissive behavior` を決める  
4. その固定値を使って青写真・エンブレム・イラストを生成する  
5. 最後に SuitSpec へ落とし、Armory Viewer で成立確認する

重要：シルエット決定までをルール化し、画像生成はその後ろに置く。  

これで「毎回別物になる」事故を抑えられる。

A-1.2.1 形状可変は「完全生成」ではなくパラメトリックにやる

スーツの形を毎回変えたい欲は正しい。  

ただし現行スタックは、**先に3DパーツとUVが存在し、そこへ生成画像をテクスチャとして流し込む**方式である。  

つまり今の本線は「画像から3Dを作る」ではなく、**「3Dありきで、入力から部位構成・形状パラメータ・UVテクスチャを決める」** である。

三面図も、現段階ではメッシュ成立前の設計資料ではなく、**組み上がったスーツ状態から再出力される確認図** と捉えるほうが正しい。

したがって、三面図から完全自由形状を毎回生成し、それを安定して3Dへ落とすのは、現段階ではコストも破綻率も高い。

現実解は次：

- ベースになる装甲骨格は 4〜6 系統に固定する
- 各系統に対して `silhouette_params` で連続変形をかける
- 三面図はその結果を元に生成する

つまり順序は **「画像から形を作る」ではなく「形を決めてから画像にする」**。

さらに現場の処理順は、次で固定するのが良い：

1. 入力文から感情軸と誓いを抽出する
2. そこから装甲系統・モジュール構成・形状パラメータを決める
3. 先に3D上でスーツを成立させる
4. 各部位UV向けのテクスチャ生成を回す
5. 必要なら完成形から三面図・青写真を再出力する

推奨パラメータ：

- shoulder_gain：肩の張り出し
- chest_mass：胸部装甲の厚み
- waist_taper：腰の絞り
- forearm_length_bias：前腕の延長感
- shin_length_bias：脚部の伸長感
- crest_height：頭部／角／クラウンの高さ
- shell_wrap：包み込み度
- edge_sharpness：エッジの鋭さ
- asymmetry：左右差

これなら「都度形が違う」は実現できるし、Viewer / XR へも落としやすい。

A-1.3 どこを固定し、どこを生成に任せるか

- 固定：感情→色相、感情→蒸着プロファイル、感情→シルエット系統、部位構成、封印位置
- 条件分岐：強度で装甲密度、制御性で発光ノイズ、主副感情差で左右差、誓いで封印紋と発光規律
- 生成：表面柄、エンブレム案、ポスター絵、ドット絵化、蒸着中の粒子ルック

「全部生成にするか？」への答えは、**ノー。前段はルール、後段は生成** が基本。

A-2. 構造化（SuitSpec生成）は「画像→自動」じゃなくても良い
MVPの現実解：

画像はAI生成（見た目の興奮）

SuitSpecは **半自動（テンプレ+選択）**で早く安定化

モジュールはキット化
生成画像は「各部位UVテクスチャ / 投影 / 刻印 / デカール」として利用

3Dメッシュそのものの完全自動生成は後回しでよい。

ただし完全固定メッシュだけだと物足りないので、当面は `silhouette_family + silhouette_params` でボディ／装甲を可変にする。  

三面図はそのパラメータ結果を描かせる側に回す。

## 1つだけ、あなたの熱を増幅させる提案（決め打ち）

「SIM精度を上げる」＝ **“鎧が立つ”**の判定基準を、最初に3つに絞りましょう（エンジニアと揉むときの共通指標になる）。

- **Silhouette（シルエットが強い）**：遠目に鎧だと分かる  

- **Joint Safety（関節が破綻しない）**：膝/肘/肩に逃げがある  

- **Seal Readability（封印が読める）**：胸の決めポイントが潰れてない  

この3つを満たす限り、生成の自由度は無限に広げてOK。  

満たさないなら「生成の良し悪し」ではなく「規格外（設計不成立）」として潔くリフォージ（再鍛造）する。
