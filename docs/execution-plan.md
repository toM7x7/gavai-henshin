# Execution Plan (2026-03-02)

## 目的

- 体に沿う装着品質を上げる
- 生成待ち時間を短縮する
- Live入力（WebCam / mocopi）へ段階的に接続する

## 現在地（完了）

1. `body-fit` に VRM 読込・骨表示・Attach切替を実装
2. three-vrm 読込経路を安定化（importmap + runtime更新）
3. VRMローダ責務を `vrm-loader.js` に分離（保守性改善）

## 進行状況（本ターン）

1. `helmet/chest` 対策として、部位別の骨フォールバック解決を追加
2. `meta` / 凡例に `missing anchor` 情報を追加して診断性を向上
3. `body-fit` / `suit-dashboard` 手順書を現行コマンドに更新
4. `body-fit` に `VRM Anchor Editor` を追加（全身プレビューで直接調整可能）
5. `attachment_slot` ベースの部位整合処理と Attach 3モード（BodySim/Hybrid/VRM）を追加
6. `body-fit` に VRM単体確認導線（Armor Off / Focus VRM / Idle motion）を追加

## フェーズA: 装着品質の安定化（最優先）

### A-1. 部位ごとの初期anchor最適化

- 対象: `helmet/chest/back/waist/thigh/boot/hand`
- 作業:
  1. `vrm_anchor` の基準値を部位別に見直し
  2. BodySim / VRM 切替時の差分を縮小
  3. `helmet/chest` の見切れ再発を抑制
- 完了条件:
  - `body-fit` で全対象部位が可視
  - `Attach: VRM` / `Attach: Hybrid` 時に大きな飛び・回転破綻がない

### A-2. フィッティング操作性の改善

- 作業:
  1. Fit編集の適用速度を改善（即時反映の安定化）
  2. 推奨値（Suggest）のロジックを部位別に調整
- 完了条件:
  - 操作後 1 秒以内に視覚反映
  - `fit_score` の平均改善

## フェーズE: 体験差別化スパイク（外部調査反映）

このフェーズは「実装を止めない前提」の検証レーン。主線（A〜D）を維持しつつ、週次で1本ずつ最小検証する。

### E-1. WebCam変身の最短化（Try-On転用）

- 狙い:
  - WebCam再変身を `Pose推定 + オクルージョン + 装着UI` の最短構成で成立させる
- 作業:
  1. MediaPipe Pose Landmarker で `body-sim` 互換フレームを安定生成
  2. 全身オクルージョンと鎧ON/OFF UIを最小実装
  3. セッションIDで「再変身ログ」を保存
- 完了条件:
  - ノートPC + WebCamで 30fps 目標（最低 24fps）を維持
  - 同じ `suit_id` で再変身リプレイ可能

### E-2. XR蒸着演出の強化（mocopi補完レーン）

- 狙い:
  - mocopi依存を避けつつ、Quest内の「全身感」不足を埋める
- 作業:
  1. Meta Movement SDK を検証導入（Body trackingの成立可否確認）
  2. `input_source=sim/webcam/mocopi/movement` 抽象を統一
  3. 失敗時フォールバック（BodySim固定）を先に実装
- 完了条件:
  - 変身シーケンス中の入力断でも演出が継続
  - 追従失敗が発生しても体験中断しない

### E-3. 鎧立て（三面図）から3Dへの接続

- 狙い:
  - 2D生成結果を3Dへ短距離で接続し、装着試作サイクルを短縮する
- 作業:
  1. `front/side/back` の3面画像を生成し、同一性を評価
  2. Hunyuan3D-2 または TripoSR でドラフトメッシュ化
  3. Unity側で Decal/Projection を使い、品質不足を演出で補正
- 完了条件:
  - 1セッション中に「三面図 -> メッシュ -> 装着確認」まで到達
  - 次回反復用に失敗パターン（破綻部位）をログ化

## 外部調査の取り込み方針

- 参考情報は `docs/reference-map-2026-03.md` に集約し、週次で更新する
- 採用判断は「実装コスト < 2日」「既存Trackと衝突しない」を満たすものに限定する
- 競合比較は「機能の有無」ではなく「プロトコル連結（生成 -> 蒸着 -> 封印 -> 再変身）」で評価する
- 生成モデルはIDの短期変動を前提に、CLIオプションと設定ファイルで即時切替できる状態を維持する
